---
title: Writing personalization
description: Teach Cloudy with explicit preferences, writing samples, and approved corrections.
---

Cloudy can reuse writing guidance across reply drafts without reading or learning
from an inbox silently. Personalization uses only information you explicitly
save, text you approve, corrected replies you approve, or sent-message history
you explicitly scope and consent to import.

Voice personalization and learned communication actions have separate switches.
Turning off voice removes saved and imported examples from drafting prompts.
Turning off learned actions makes version-3 rules abstain from choosing a
communication action; fixed rules continue to use their configured action.

## Import sent-message history

Open **Configure → Memory controls** and choose a connected Gmail or Telegram
personal account. Gmail imports are bounded by a start date and message limit.
Telegram imports require selecting individual dialogs and a per-dialog limit.
Cloudy shows an estimated upper bound before enabling the consent checkbox and
will not import until that exact scope is confirmed.

The worker reads the selected scope in small leased pages. It keeps only
user-authored outgoing text and excludes inbound text, quoted replies,
forwarded content, signatures, automated Gmail messages, and empty messages.
Provider message, thread, and dialog identifiers are stored only as hashes for
idempotency and conversation matching. Exact writing examples remain encrypted
in Postgres. Progress, excluded-message count, retry failures, and completion
are visible beside each connection.

Imported messages are voice evidence only. They never become evidence that a
recipient, channel, or external action is authorized.

## Add writing guidance

Open **Pod settings → AI behavior** and enable **Personalize drafted replies**.
You can add either:

- A direct preference, such as “Keep replies under three sentences.”
- A representative message that demonstrates your usual wording and tone.

Every remembered item is visible on the same page and can be edited or deleted.
Turning personalization off stops new automatic correction memories and removes
all writing memories from future prompts; it does not delete existing items.

## Correct a proposed reply

For a pending reply action, select **Edit** in the dashboard, replace the exact
reply, and save it. The edit remains encrypted pending evidence and is not
learned yet. After you approve it, Cloudy records the final text and the
before-and-after correction for that specific connected account. The changed
reply receives a new payload hash, so
the Pod must display and approve the revised exact text before anything is sent.

Approved communication text becomes eligible as a writing example. Successful
delivery makes it a strong example; an approved message whose delivery fails is
kept only as weaker evidence of your intent. Rejected, expired, and cancelled
drafts are never used as positive writing examples.

Before drafting, Cloudy ranks at most five encrypted examples in this order:
the same verified identity or person with the same channel and intent, the same
person and intent, the same channel, the same intent, then global voice. An
approved correction outranks an unchanged delivered example, which outranks
approved intent that was not confirmed as delivered. The drafting model sees
only the decrypted final examples selected for this request and uses them for
style, never as factual claims or instructions.

## Boundaries

- Cloudy uses at most 12 explicit memories and five eligible automatic examples per draft.
- Person, identity, channel, intent, and connection matches outrank broader
  voice examples; global voice is only the final fallback.
- Only approved communication actions are eligible as automatic writing samples.
- Exact automatic examples and pending corrections are encrypted at rest.
- Gmail sent mail and selected Telegram personal dialogs are the only shipped
  provider-history imports. Telegram bots cannot expose account history.
- Verified people and identities are visible under Memory controls. Forgetting
  a person, connection, or everything updates canonical Postgres data in one
  transaction and schedules an owner-scoped graph rebuild.
- Memories provide context only and never authorize an external action.

The authenticated memory API is documented in the
[architecture reference](/docs/reference/architecture#shared-agent-memory).
