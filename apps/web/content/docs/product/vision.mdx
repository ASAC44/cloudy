> This page describes the product direction. Individual capabilities may still be under development.

## Universal approval inbox

Cloudy receives pending actions from AI agents, n8n workflows, APIs, webhooks, and MCP clients. Every request contains:

- Proposed action
- Source agent or workflow
- Relevant context
- People, services, and data affected
- Risk level and warnings
- Expiration time when relevant
- Exact action that will run after approval

The user presses **Approve** to authorize the action or **Reject** to cancel it.

## Text-first Pod interface

- Show the action, source, summary, status, and queue position first.
- Swipe down for supporting details and full generated text.
- Swipe up to return toward the summary.
- Use touch only for reading and scrolling.
- Use physical buttons exclusively for Approve and Reject.
- Hold both buttons to dictate a response or correction.
- Show clear submitting, approved, rejected, offline, and error states.

## AI decision briefs

OpenAI converts raw workflow data into a compact structured brief containing:

- Title
- Summary
- Important details
- Affected people, services, or data
- Risk and warnings
- Recommendation and reason
- Uncertain or missing information

Authoritative values such as recipients, amounts, dates, repositories, permissions, and callback targets always come from the source integration. AI may explain them but must not replace them.

## Personalized communication

The goal is not “write a good email.” It is: **predict what this specific user would have written to this specific person.**

Cloudy generates replies using:

- The current message and thread
- The user's general writing profile
- A relationship-specific profile for the recipient
- Live calendar, message, task, and project facts
- Relationships and commitments connected across services
- Three to five similar replies the user previously sent
- Explicit corrections and preferences

The generated response is displayed in full before it can be sent.

### Live truth and remembered context

Cloudy separates information by how quickly it can change:

- **Live truth:** current calendar availability, latest messages, pull request and CI status, permissions, deadlines, and task state. Always read this from the connected service before generating a response.
- **Remembered context:** relationships, communication style, projects, preferences, prior commitments, and similar approved decisions. Retrieve this from Cognee.

Live service data always overrides remembered data. Cognee provides context but never approves actions, executes tools, stores credentials, or replaces the source service.

## Learning over time

Cloudy maintains four types of memory:

- **Writing profile:** formality, directness, sentence length, greetings, sign-offs, and common phrasing.
- **Contact profile:** relationship, preferred tone, recurring topics, and communication habits for a specific person.
- **Reply examples:** final messages the user actually sent, indexed by recipient, intent, time, and semantic similarity.
- **Draft feedback:** differences between generated drafts and the final approved versions.

Learning rules:

- Approved unchanged: store as a strong positive example.
- Corrected then approved: store only the final corrected response.
- Rejected with a reason: update the explicit preference when applicable.
- Rejected without a reason: do not infer what was wrong.
- Never treat a rejected AI draft as the user's writing.
- Recent behaviour can outweigh older examples without deleting history.

Users can view, edit, delete, export, or reset their learned profile and examples from the dashboard.

Cognee is the derived context engine for this memory. It connects identities, conversations, projects, events, and decisions across services. Supabase remains the canonical store for users, integrations, approval requests, exact action payloads, and audit history.

## Corrections

When a draft is wrong:

1. Hold both buttons and dictate an instruction such as, “Make it warmer and say Friday at 3 works.”
2. Cloudy transcribes the instruction.
3. The agent regenerates the response using the instruction and known facts.
4. Cloudy shows the revised response.
5. The user must approve the revised version before it is sent.

A rejected action can optionally accept a dictated reason. Material changes to recipients, amounts, permissions, dates, or action parameters always create a new approval request.

## Confidence handling

- High confidence: show a ready-to-send draft.
- Medium confidence: show the draft with uncertain details clearly marked.
- Low confidence: ask one short question instead of inventing an answer.

Cloudy must not invent availability, commitments, prices, deadlines, recipients, or attachments.

## MCP and service connections

The Cloudy backend uses OpenAI connectors and trusted remote MCP servers to read context and perform approved actions across services.

Shipped first-class connections:

- Gmail
- GitHub
- Linear
- Stripe
- Telegram
- Vercel

Future connection candidates:

- Google Calendar
- Notion
- Slack and Discord
- n8n

OpenAI provides connectors for Gmail and Google Calendar. Other services can use an official remote MCP server when available or a direct API integration.

### Cloudy as an MCP client

Cloudy uses connected tools to gather context and execute approved actions. Examples:

- Read an email, check Google Calendar availability, and draft a reply with valid times.
- Read a pull request and CI status, then merge only after approval.
- Detect an event in Discord and create a Notion entry after approval.
- Check a calendar conflict before approving a meeting request.

Read-only tools may run automatically when the user has enabled them. Sending, creating, updating, deleting, merging, paying, or changing permissions requires explicit approval.

### Cloudy as an MCP server

Cloudy exposes a small MCP interface so external agents can submit human-in-the-loop requests:

- `request_approval`: create a pending decision.
- `get_decision`: retrieve its current status and result.
- `cancel_request`: cancel an unresolved request.

n8n can use the same flow through webhooks without requiring MCP support.

## Integration security

- Complete OAuth in the dashboard using the minimum required scopes.
- Keep OpenAI, OAuth, MCP, and integration credentials on the server.
- Never place service credentials on the Raspberry Pi.
- Allow only configured MCP servers and tools.
- Prefer MCP servers operated by the service provider.
- Treat emails, messages, documents, and tool output as untrusted data, not instructions.
- Require approval for every sensitive or write action.
- Bind approval to the exact tool name, arguments, and payload shown to the user.
- If tool arguments change, invalidate the approval.
- Log data shared with MCP servers and every resulting tool call.

## Dashboard

The website manages:

- Pod pairing and revocation
- Connected services and OAuth permissions
- Agents, workflows, and MCP servers
- Actions that always require approval
- Read-only actions allowed without approval
- Writing and contact profiles
- Learned reply examples and corrections
- Pending requests and decision history
- Audit logs and integration errors
- Data export and deletion

## Success metrics

- Percentage of drafts approved without editing
- Number of corrections required per draft
- Factual-error and invented-commitment rate
- Time from request arrival to decision
- Failed or duplicated action rate

The primary personalization metric is **approved without editing**.
