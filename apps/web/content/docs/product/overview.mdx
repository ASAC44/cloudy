title: How Cloudy works
description: The core human-in-the-loop interaction, hardware, and product principle.
---

Cloudy is a pocket-sized human-in-the-loop Pod for approving, rejecting, or responding to actions proposed by AI agents and automated workflows without opening a phone or computer.

The public landing page includes an interactive, downloadable Excalidraw system
map that follows this complete path from connected source through Cloudy,
Supabase, the keychain Pod, and the exact approved action.

## Core interaction

An agent prepares an action and sends a decision request to Cloudy. The Pod shows the relevant context on its 2.8-inch screen. The user can:

- Press **Approve** to authorize the action.
- Press **Reject** to stop the action.
- Hold both buttons to dictate a response or instruction through the microphone.

After approval, either the connected agent resumes or Cloudy's guarded API runtime delivers the exact configured provider action or workflow callback. Cloudy remains the authorization boundary: model output alone never permits execution.

## Pod hardware

- Small cuboid keychain form
- Raspberry Pi Zero 2 W
- 2.8-inch screen
- Approve and Reject buttons
- Microphone for dictation
- No speaker
- Wireless network connection

## Decision details

Each request shows enough information to make a safe decision, including:

- The proposed action
- The requesting agent or workflow
- A concise summary and supporting details
- The people, services, or data affected
- Risk, warnings, and expiration time when relevant

## Shipped integrations

The current release accepts approval requests through automation keys and supports GitHub, Gmail, Google Calendar, Vercel, Telegram, Linear, Stripe, Notion, and trusted remote MCP connections. A local Codex bridge can turn dictated Pod tasks into read-only plans and route implementation approvals back to the device.

Broader n8n, Discord, WhatsApp, Slack, and universal MCP workflows are
part of the clearly labeled [Product vision](/docs/product/vision), not the
current compatibility list.

Example actions include:

- Approving and merging a pull request
- Reviewing and sending a drafted email reply
- Suggesting available meeting times from Google Calendar in an approved Gmail reply
- Adding a detected event to Notion
- Authorizing a deployment or workflow step
- Rejecting an unwanted or risky action

## Personalized communication

The goal is not “write a good email.” It is: **predict what this specific user would have written to this specific person.**

The current release combines live provider context with explicit remembered
context. Users can save writing preferences and samples, or edit a proposed
reply in the dashboard. An edit remains encrypted and pending until the user
approves the action. Cloudy then reuses the approved communication as a writing
example for that connected account, weighting successful delivery more strongly
than approved intent that could not be delivered.

Cloudy does not silently import sent mail, and rejection is never treated as a
positive writing example. See [Writing personalization](/docs/product/personalization)
for the shipped flow and [Product vision](/docs/product/vision) for planned
relationship-aware profiles and broader context.

Rules can also explicitly opt into learned communication selection. This is
limited to pre-authorized Gmail and Telegram candidates; existing rules stay
fixed. Cloudy abstains when graph evidence is unavailable or weak, and every
selected recipient, channel, and exact message is still shown for Pod review.

## Dashboard

The web dashboard pairs Pods and Codex bridges, manages connections and AI
settings, assigns up to six feeds to each of three screens with a drag-and-drop
Kanban board, builds dynamic Ping rules, issues automation keys, and shows
decisions and rule activity. Moving an app attaches it to the destination
without replacing its existing app subscriptions. Screen 2 is the default;
horizontal swipes open Screens 1 and 3. Pings from all apps attached to a screen
share the same queue instead of being grouped by app, so GitHub, Gmail, and
GitHub arrivals appear in that sequence. Swipe-down reveals the active Ping's
details. Any screen without an active Ping shows Cloudy immediately.

When the paired Pod is online, **Play with Cloudy** on Home can trigger a
one-shot Blink, Yawn, Sleep, or Jump mascot animation. The animation briefly
takes over the display, then returns to the current screen without clearing a
pending approval.

## Product principle

Agents do the work. Cloudy keeps the user in control at the moment a decision matters.
