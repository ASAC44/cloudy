---
title: How Podex works
description: The core human-in-the-loop interaction, hardware, and product principle.
---

Podex is a pocket-sized human-in-the-loop Pod for approving, rejecting, or responding to actions proposed by AI agents and automated workflows without opening a phone or computer.

## Core interaction

An agent prepares an action and sends a decision request to Podex. The Pod shows the relevant context on its 2.8-inch screen. The user can:

- Press **Approve** to authorize the action.
- Press **Reject** to stop the action.
- Hold both buttons to dictate a response or instruction through the microphone.

After approval, either the connected agent resumes or Podex's guarded API runtime delivers the exact configured provider action or workflow callback. Podex remains the authorization boundary: model output alone never permits execution.

## Pod hardware

- Small cuboid keychain form
- Raspberry Pi Zero 2 W
- 2.8-inch screen
- Approve and Reject buttons
- Microphone for dictation
- No speaker
- Wireless network connection

## Decision details

Each request shows enough information to make a safe decision, including:

- The proposed action
- The requesting agent or workflow
- A concise summary and supporting details
- The people, services, or data affected
- Risk, warnings, and expiration time when relevant

## Shipped integrations

The current release accepts approval requests through automation keys and supports GitHub, Gmail, Google Calendar, Vercel, Telegram, Linear, Stripe, Notion, and trusted remote MCP connections. A local Codex bridge can turn dictated Pod tasks into read-only plans and route implementation approvals back to the device.

Broader n8n, Discord, WhatsApp, Slack, and universal MCP workflows are
part of the clearly labeled [Product vision](/docs/product/vision), not the
current compatibility list.

Example actions include:

- Approving and merging a pull request
- Reviewing and sending a drafted email reply
- Suggesting available meeting times from Google Calendar in an approved Gmail reply
- Adding a detected event to Notion
- Authorizing a deployment or workflow step
- Rejecting an unwanted or risky action

## Personalized communication

The goal is not “write a good email.” It is: **predict what this specific user would have written to this specific person.**

The current release combines live provider context with explicit remembered
context. Users can save writing preferences and samples, or edit a proposed
reply in the dashboard. Corrections are stored for that connected provider
account and reused in later drafts.

Podex does not silently import sent mail or learn from ordinary approvals. See
[Writing personalization](/docs/product/personalization) for the shipped flow
and [Product vision](/docs/product/vision) for planned relationship-aware
profiles and broader context.

## Dashboard

The web dashboard pairs Pods and Codex bridges, manages connections and AI
settings, assigns one feed to each of three screens with a drag-and-drop Kanban
board, builds dynamic Ping rules, issues automation keys, and shows decisions
and rule activity. Screen 2 is the default; horizontal swipes open Screens 1
and 3. A new notification opens its assigned feed and swipe-down reveals its
details. Screen 2 remains visible by default; the mascot appears only after an
inactivity delay.

## Product principle

Agents do the work. Podex keeps the user in control at the moment a decision matters.
