---
title: Codex bridge
description: Pair a local Codex installation, expose selected workspaces, and send voice tasks from the Pod.
---

The bridge connects Podex to a local Codex installation without uploading
repository paths, Codex credentials, or source files to Podex. It talks to
`codex app-server` over local standard input and output, while the hosted API
receives opaque workspace identifiers, labels, session status, and approval
requests.

## Prerequisites

- Node.js 20 or newer
- Codex CLI 0.144.5 or newer, already signed in on the local computer
- A running Podex API and access to the authenticated **Codex** dashboard page

## Install and pair

From the repository root:

```sh
cd apps/bridge
npm link
podex-bridge pair http://localhost:3001
```

For a hosted deployment, replace the local URL with the public Podex API
origin. Enter the eight-character code printed by the command on the
dashboard's **Codex → Pair bridge** dialog.

The bridge stores its token and local workspace paths in
`~/.config/podex/bridge.json` with mode `0600`. The token is scoped to that
bridge and can be revoked from the dashboard.

## Add a workspace

```sh
podex-bridge add-workspace /absolute/path/to/repository
podex-bridge status
```

Choose the workspace and an existing or new session on the Codex dashboard.
The selected target is where the Pod sends its next dictated task.

Run the bridge in the foreground while testing:

```sh
podex-bridge run
```

Once it is working, install the current user service on macOS or Linux:

```sh
podex-bridge install-service
```

`uninstall-service` removes the background service but keeps the bridge
configuration. `remove-workspace` stops exposing one local repository.

## Pod workflow

Hold both Pod buttons, speak a task, and release them. Podex uploads a mono
16 kHz WAV recording to the API, shows the transcript, and sends it only after
you approve the transcript. Codex first plans in a read-only sandbox. The plan
returns to the Pod for approval; implementation starts with workspace write
access only after that approval.

Command, file-change, and permission requests raised during implementation
also return to the Pod. The bridge rejects unsupported request types and
refuses an approved plan if the repository changed after the plan was sealed.

Voice transcription requires the official OpenAI provider and API base URL in
**Settings**. Other supported AI providers can be used by the rule builder,
but they do not enable Pod voice transcription.

## Troubleshooting

- `Codex CLI is not installed`: ensure `codex --version` works for the same
  user that runs the bridge.
- `Codex 0.144.5 or newer is required`: update the local Codex CLI.
- Bridge shown as offline: run `podex-bridge run` and inspect its error, then
  verify the API URL and the bridge has not been revoked.
- No workspace on the dashboard: add an absolute repository path and allow the
  next two-second bridge sync to complete.
