title: Service connections
description: Connect Google Calendar, Linear, Stripe, OAuth providers, and trusted remote MCP servers safely.
---

Podex stores connection credentials encrypted with `CONNECTION_ENCRYPTION_KEY`.
The dashboard never receives a saved key after submission, and replacing a key
runs a fresh connection test.

## Connect Gmail

Gmail watch rules first receive stable message and thread IDs, then Podex reads
the matching thread through the same read-only OAuth connection before creating
the encrypted Pod notification. The summary shows the sender and subject;
swipe down on the Pod to read the complete email. Reply rules additionally show
the exact proposed response. Existing inbox messages form the first polling
baseline, so send a new message after activating a rule when testing.

## Connect Google Calendar

Enable the Google Calendar API and register the
`/v1/connections/oauth/google_calendar/callback` URL as described in
[Authentication setup](/docs/development/authentication). Then choose
**Google Calendar** on the Connections page and continue with Google OAuth.

When using `./start.sh`, Google Calendar OAuth state and the encrypted Calendar
connection are stored in `apps/api/.state/podex.sqlite`. This narrow local
fallback supports connecting and retesting Calendar without local Supabase;
creating active Calendar Pings still requires the tracked Supabase migration.

Calendar rules can list available calendars and read events only inside an
explicit RFC 3339 time window. Event creation and updates are actions: Podex
shows the exact title, times, timezone, location, description, and attendees
for Pod approval before calling Google. Updates also include the source event
etag and fail if the event changed after approval. Deleting events is not
exposed. Google Calendar uses the narrowly scoped `calendar.events` and
`calendar.calendarlist.readonly` permissions documented in
[Google's Calendar scope reference](https://developers.google.com/workspace/calendar/api/auth).

## Connect Linear

1. In Linear, open **Settings → Account → Security & Access** and create an API
   key for Podex. Prefer the narrowest access available for the workspace and
   actions you plan to use.
2. In Podex, open **Connections**, choose **Linear**, name the connection, and
   paste the key.
3. Select **Save & test**. Podex connects only to Linear's official Streamable
   HTTP endpoint at `https://mcp.linear.app/mcp` and reports the number of tools
   it discovers.

See [Linear's MCP documentation](https://linear.app/docs/mcp) for current key
and server requirements.

## Connect Stripe

Start in a Stripe sandbox. In **Developers → API keys**, create a restricted
key that grants only the resources and operations required by the Ping. Do not
use a publishable key, and avoid unrestricted or live keys during setup.

Choose **Stripe** on the Podex Connections page and paste the restricted key.
Podex connects only to `https://mcp.stripe.com`. Stripe controls which tools are
available from the permissions on that key. See [Stripe's MCP documentation](https://docs.stripe.com/mcp)
and [key security guidance](https://docs.stripe.com/keys-best-practices).

## Connect Telegram

Choose **Personal account** to scan Telegram's QR code. This mode watches DMs,
groups, and channels through the worker and requires `TELEGRAM_API_ID` plus
`TELEGRAM_API_HASH` on both the API and worker services.

Choose **Bot** to paste a BotFather token. After `getMe` succeeds, Podex
registers an HTTPS webhook under `PODEX_PUBLIC_API_URL` with Telegram's secret
token header. Incoming text, captions, and attachment metadata enter the same
deduplicated Ping engine. A bot reply is available only as an exact Pod-approved
action; an uncertain Bot API delivery is not retried automatically. Retest an
existing bot connection after deploying this version to register its webhook.
See Telegram's [Bot API webhook documentation](https://core.telegram.org/bots/api#setwebhook).

For local bot testing, run `./scripts/run-ngrok.sh`, copy its HTTPS forwarding
URL into `PODEX_PUBLIC_API_URL` in `apps/api/.env`, and restart Podex before
connecting or retesting the bot. Keep ngrok running while Telegram sends
webhook updates.

## Capability safety

Podex follows MCP tool annotations instead of guessing what a tool does:

- Tools explicitly marked read-only may provide source or context data.
- Tools explicitly marked non-destructive writes may become rule actions, but
  the exact payload must be approved on the Pod before execution.
- Destructive tools are excluded.
- Unannotated tools are shown as unavailable at runtime.

Provider permissions remain an independent safety boundary. Give each key only
the access its rule needs even when Pod approval is enabled.

## Rotate or revoke a key

Use **Replace token** on the connection, paste the new key, and let the smoke
test finish. Then revoke the old key in Linear or Stripe. If a key might have
been exposed, revoke it at the provider first and replace it in Podex
immediately. Disconnecting a provider deletes its encrypted credential from
Podex but does not revoke the key at the provider.

GitHub, Gmail, and Google Calendar use OAuth. Vercel uses a token, while Telegram
supports the bot webhook and personal-account flows above. **Custom MCP** uses
public HTTPS Streamable HTTP endpoints: Podex discovers annotated tools, polls
verified read tools as sources, and runs verified non-destructive writes only
after exact Pod approval.
