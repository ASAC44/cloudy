---
title: Develop the Pod locally
description: Run and test the 640×480 Pod simulator without Raspberry Pi hardware.
---

The Pod application uses a 640×480 landscape pygame renderer on macOS and the
Raspberry Pi. Development mode adds only a virtual enclosure, clickable
buttons, and desktop input mappings.

## Setup

The repository setup script installs the web and API dependencies and creates
`apps/pod/.venv` with Python 3.13:

```sh
./scripts/setup.sh
```

Create the documented environment files before starting all three apps:

- `apps/api/.env`: `SUPABASE_URL`, the server-only `SUPABASE_SECRET_KEY`, and
  the connection variables from `.env.example`. Generate the local encryption
  value with `openssl rand -base64 32` and keep it stable.
- `apps/web/.env.local`: the two `NEXT_PUBLIC_SUPABASE_*` values and
  `CLOUDY_API_URL=http://localhost:3001`.
- The Pod defaults to the local API and stores simulator state under
  `apps/pod/.state` when launched through the root script.

Apply the tracked migration to the linked hosted development project:

```sh
supabase link
supabase db push
```

Do not edit the hosted schema manually after adopting migrations.

## Run

```sh
./start.sh
```

The root script also starts the dedicated rule worker, so active Pings are polled while the simulator is running.

The virtual Pod displays an eight-character code. Sign in to the dashboard,
enter that code, and create a **New Ping**. After pairing, the Pod holds an
authenticated Server-Sent Events connection. An event wakes the Pod to fetch
the authoritative screen snapshot; decisions and voice uploads remain ordinary
HTTP requests. If realtime delivery is unavailable, the Pod polls every 10
seconds until it reconnects, with a 90-second safety refresh while connected.
The dashboard can send realistic General, GitHub, Deployment, Gmail, or Codex mock reviews
to any of the three Pod screens. GitHub includes checks, review evidence, and
rollback facts; Gmail shows the original message and exact drafted reply; Codex
shows a brief plan summary, with the numbered implementation plan available in
Details. On the Codex mock, hold both Pod buttons
to dictate a revision, confirm the transcript, and return to the plan before
approving or rejecting it. The revision is demo-only and stays on the Pod; real
Codex work continues to require a paired bridge and selected target.
The dashboard control strip can approve or reject its oldest pending Ping next
to the screen navigation buttons; it uses the same atomic, hash-bound decision
path as the Pod controls.
Both dashboard and Pod decisions play distinct approval or rejection animations
on the physical display before Cloudy returns to idle.

## Pitch console

Open `/demo` after signing in to run a presentation from one page. The console
shows Pod connectivity and the pending count, sends any realistic mock to a
chosen screen, triggers Cloudy’s four mascot moves, navigates between screens
or notification details, scrolls long details up or down in visible steps, and approves
or rejects the current Ping. These
controls call the same authenticated actions as Home; there is no separate
demo-only API or decision path. Rapid navigation presses are delivered to the
Pod in order. Swipe Up and Swipe Down enter and leave notification details;
the separate Scroll Up and Scroll Down controls only move content while the
details page is open. Notification summaries use large decision text and keep warnings
on the scrollable details page so the primary action stays readable at a glance.

Simulator controls:

- Drag inside the screen to scroll. The simulator renders the Pod screen at
its native 640×480 resolution for a sharp preview.

For a browser-only walkthrough, open **Settings → Presentation → Demo mode**.
Home then adds a Sentry and Datadog example to the Ping chat. Selecting it shows
both observability sources as already connected and walks through anomaly
detection to review; it does not call the API or create a real Ping.
- Click the illustrated buttons or press `A` and `R` to decide.
- Press `V` once to start voice recording and again to stop it. On hardware,
  hold both physical buttons for 600 ms and release either button to stop.
- Press `O` to simulate a disconnection. Cached content remains readable, but
  decisions stay disabled until the connection returns.
- Screen 2 is the default feed. Swipe left for Screen 1 and right for Screen 3;
  each screen receives Pings from up to six apps configured on the dashboard
  Kanban board. Apps route Pings but are not grouped or shown as feed rows.
- Swipe up from a feed to open Quick Settings. Drag or tap the brightness and
  volume sliders; tap the other rows to toggle connectivity or reduced motion.
  Settings persist locally; swipe down to return to the feed.
- A new notification opens its assigned screen. Swipe down to see its details
  and swipe up to return to the summary. Reject removes the notification.
- Pairing, approval, voice, result, and error states keep their dedicated
  screens. Any screen without an active Ping shows Cloudy's ambient animation.
- Press the down and up arrow keys to move between a notification summary and
  its details, or to close and open Quick Settings from a feed.
- Press Escape to close the simulator.

Test Pings record only the decision. They never execute an external action.
Voice transcription requires the official OpenAI provider to be configured in
the dashboard and a selected [Codex bridge target](/docs/development/codex-bridge).

## Checks

```sh
./scripts/check.sh
./scripts/check-pod-arm.sh
```

The second command is optional and requires Docker. It installs and tests the
Pod under Linux ARM64 with the Pi's 512 MB memory ceiling. It does not emulate
the display controller, GPIO wiring, microphone, Wi-Fi, or CPU timing.

Pairing credentials are written with mode `0600`. Delete `apps/pod/.state` to
reset only local simulator pairing; use **Unpair Pod** in the dashboard to
revoke the server-side token.
