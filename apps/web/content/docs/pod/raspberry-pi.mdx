## Provision the Pi

Flash Raspberry Pi OS Lite 64-bit with Raspberry Pi Imager. Configure Wi-Fi,
hostname, a `cloudy` user, and public-key SSH access in Imager. After boot:

```sh
sudo apt update
sudo apt full-upgrade
sudo apt install alsa-utils cage python3-venv python3-pip
sudo usermod -a -G input,render,video,gpio,audio cloudy
sudo mkdir -p /opt/cloudy-pod /var/lib/cloudy-pod
sudo chown -R cloudy:cloudy /opt/cloudy-pod /var/lib/cloudy-pod
python3 --version # must report Python 3.13.x
```

Use the current 64-bit Raspberry Pi OS Lite image that provides Python 3.13;
do not silently deploy this runtime on an older Python release.

Configure the selected HDMI/KMS display and I²C capacitive touch controller
using its current Raspberry Pi OS instructions. Do not install a vendor OS
image or legacy framebuffer-copy stack when a KMS/DRM driver is available.

### SPI ILI9341 and XPT2046 prototype

For a write-only 320×240 ILI9341-compatible panel that has no usable DRM
overlay, Cloudy can scale its native 640×480 renderer directly into the Linux
RGB565 framebuffer. With LCD CS on SPI0 CE0, D/C on GPIO25, reset on GPIO24,
touch CS on SPI0 CE1, and touch IRQ on GPIO17, add:

```ini title="/boot/firmware/config.txt"
dtparam=spi=on
dtoverlay=fbtft,spi0-0,ili9341,reset_pin=24,dc_pin=25,rotate=90,speed=16000000
dtoverlay=ads7846,cs=1,penirq=17,penirq_pull=2,xohms=100,pmax=255
```

Then run the Pod without Cage:

```sh
sudo sh -c 'TERM=linux setterm --cursor off < /dev/tty1 > /dev/tty1'
CLOUDY_SIMULATOR=0 \
CLOUDY_FRAMEBUFFER=auto \
CLOUDY_TOUCH_DEVICE=auto \
CLOUDY_BUTTONS=0 \
.venv/bin/python -m cloudy_pod
```

For a persistent SPI deployment, install the direct framebuffer service
override during the service setup below.

Quick Settings can drive a kernel-exposed backlight through the standard Linux
backlight class. Configure the display overlay's `led_pin` with a free GPIO,
then verify that the driver creates `/sys/class/backlight/<device>`. Do not use
GPIO18 for the backlight because the INMP441 uses it for the I²S clock. Grant the
`cloudy` service user write access to that device's `brightness` file, then set
either automatic discovery or its exact directory in `/etc/cloudy-pod.env`:

```dotenv
CLOUDY_BACKLIGHT=auto
# CLOUDY_BACKLIGHT=/sys/class/backlight/<device>
```

An explicitly configured invalid device stops startup with a clear error. If a
working device later rejects a write, the Pod keeps running and uses software
dimming until its next restart. The Raspberry Pi `fbtft` overlay documents the
configurable [`led_pin`](https://github.com/raspberrypi/firmware/blob/master/boot/overlays/README).

Touch coordinates come from the kernel-reported axis ranges. If the physical
orientation differs, set `CLOUDY_TOUCH_SWAP_XY=1`, `CLOUDY_TOUCH_INVERT_X=1`, or
`CLOUDY_TOUCH_INVERT_Y=1` as needed. The XPT2046 interrupt wire is required;
without `T_IRQ` connected to GPIO17, Linux exposes the device but reports no
touches. Omit `CLOUDY_BUTTONS=0` after wiring the GPIO5/6 approval buttons and
installing a supported gpiozero pin backend.

### Four-leg approval buttons

A four-leg tactile switch contains two internally connected pairs; pressing the
button connects the two pairs. Confirm the pairs with a continuity meter, then
wire one leg from one pair to the GPIO and one leg from the opposite pair to
ground. Leave the other two legs unused.

| Action | BCM GPIO | Header pin | Other switch side |
| --- | --- | --- | --- |
| Accept | GPIO5 | 29 | Shared GND, pin 30 |
| Reject | GPIO6 | 31 | Shared GND, pin 30 |

The runtime enables internal pull-ups, so a press connects the input to ground.
Do not connect either button to 3.3 V. A single-button release decides; holding
both buttons for the configured threshold opens the microphone, and releasing
either stops recording. A short two-button press is ignored.

### INMP441 I²S microphone

Wire one mono INMP441 on the left I²S channel:

| INMP441 | Raspberry Pi Zero 2 W |
| --- | --- |
| VDD | 3.3 V, header pin 1 |
| GND | GND, header pin 6 |
| L/R | GND for the left channel |
| SCK | BCM GPIO18, header pin 12 |
| WS | BCM GPIO19, header pin 35 |
| SD | BCM GPIO20, header pin 38 |

The deployment includes an idempotent provisioner that installs `alsa-utils`,
adds the `cloudy` user to the `audio` group, and adds the precompiled
capture-card overlay when missing:

```sh
sudo sh /opt/cloudy-pod/deploy/provision-inmp441.sh --apply
```

Exit status `10` means the overlay was added but is not active until reboot.
The normal deployment command handles that status and schedules the one
required reboot automatically. The resulting boot setting is:

```ini title="/boot/firmware/config.txt"
dtoverlay=googlevoicehat-soundcard
```

After reboot, the same provisioner verifies the active card, service audio
permissions, exact WAV contract, file size, and a nonzero three-second signal:

```sh
sudo sh /opt/cloudy-pod/deploy/provision-inmp441.sh --check
```

For manual diagnosis, inspect the named card or make a short recording directly:

```sh
arecord -l
arecord -D plughw:CARD=sndrpii2scard,DEV=0 -q \
  -f S16_LE -r 16000 -c 1 -d 3 /tmp/cloudy-mic.wav
file /tmp/cloudy-mic.wav
```

The overlay captures at fixed 48 kHz/32-bit; `plughw` converts it to Cloudy's
mono 16 kHz/16-bit upload contract. The production service uses this named card
by default, avoiding unstable numeric ALSA card indexes. Override
`CLOUDY_MIC_DEVICE` only when `arecord -l` reports another name, such as for a
USB microphone or an ALSA `softvol` PCM added after measured input is too quiet.

See the [Raspberry Pi I²S pin map](https://pip-assets.raspberrypi.com/categories/1259-audio-camera-and-display/documents/RP-009699-WP-1-Using%20the%20I2S%20peripherals%20on%20Raspberry%20Pi%20SBCs.pdf),
[INMP441 data sheet](https://product.tdk.com/system/files/dam/doc/product/sw_piezo/mic/mems-mic/data_sheet/inmp441.pdf),
and [Bookworm-compatible overlay setup](https://learn.adafruit.com/adafruit-i2s-mems-microphone-breakout/raspberry-pi-wiring-test).

## Copy and install

For an already-provisioned Pod, deploy runtime updates from the repository root
with one command:

```sh
./scripts/deploy-pod.sh
```

Pass a different SSH destination when needed:

```sh
./scripts/deploy-pod.sh cloudy@192.168.1.50
```

The script syncs the Pod runtime, deployment files, and dependency manifests;
provisions the INMP441 overlay and audio permissions; reinstalls the Python
package only when those manifests change; refreshes and enables the systemd
unit; and fails if the service does not return to an active state. The first
deployment schedules one reboot when the I²S overlay is newly added or not yet
loaded. It does not overwrite the Pod token, settings, or `/etc/cloudy-pod.env`.

Use the manual steps below for the first installation.

From the development computer:

```sh
rsync -az apps/pod/ cloudy@cloudy.local:/opt/cloudy-pod/
ssh cloudy@cloudy.local '
  cd /opt/cloudy-pod &&
  python3 -m venv .venv &&
  .venv/bin/python -m pip install -c requirements.lock .
'
scp apps/pod/deploy/cloudy-pod.service cloudy@cloudy.local:/tmp/
scp apps/pod/deploy/cloudy-pod-spi.conf cloudy@cloudy.local:/tmp/
```

On the Pi, create `/etc/cloudy-pod.env`:

```dotenv
CLOUDY_API_URL=https://your-api-domain.example
CLOUDY_STATE_DIR=/var/lib/cloudy-pod
CLOUDY_APPROVE_PIN=5
CLOUDY_REJECT_PIN=6
CLOUDY_BUTTON_HOLD_SECONDS=0.6
CLOUDY_MIC_DEVICE=plughw:CARD=sndrpii2scard,DEV=0
# Set after the display driver exposes a writable Linux backlight device.
# CLOUDY_BACKLIGHT=auto
```

Then install and start the kiosk service:

```sh
sudo sh /opt/cloudy-pod/deploy/provision-inmp441.sh --apply
sudo install -m 0644 /tmp/cloudy-pod.service /etc/systemd/system/cloudy-pod.service
sudo chmod 0600 /etc/cloudy-pod.env
```

Reboot before continuing if the provisioner exits with status `10`.

SPI framebuffer installations additionally run:

```sh
sudo mkdir -p /etc/systemd/system/cloudy-pod.service.d
sudo install -m 0644 /tmp/cloudy-pod-spi.conf \
  /etc/systemd/system/cloudy-pod.service.d/spi.conf
```

Finish the installation and start the service:

```sh
sudo systemctl daemon-reload
sudo systemctl enable --now cloudy-pod
journalctl -u cloudy-pod -f
```

The service runs Cage directly on tty1, restarts after failures, and launches
the same pygame code used by the simulator. The Pod API token remains on the
Pi; Supabase and integration secrets never do.

The SPI override replaces Cage with direct RGB565 framebuffer output. Omit the
override for a KMS/DRM display. Paired Pods receive metadata-only realtime
invalidations over SSE and then fetch a complete authenticated snapshot. A
10-second degraded poll and 90-second connected safety refresh cover proxy,
Wi-Fi, or Realtime interruptions.

Swipe up from a feed to open Quick Settings. Changes persist in
`CLOUDY_STATE_DIR/settings.json`. A configured `CLOUDY_BACKLIGHT` controls
physical brightness through Linux sysfs; otherwise the Pod retains software
dimming. Volume uses the native ALSA `Master` control through `amixer`.
Connectivity and reduced motion are configurable from the same touchscreen
list.

## Hardware acceptance

Before treating the image as production-ready, verify:

- Native 640×480 KMS output or scaled 320×240 SPI output, including color,
  physical backlight brightness, rotation, and screen blanking.
- Touch coordinates, swipe direction, and physical text/button legibility.
- GPIO5 Approve and GPIO6 Reject wiring, pull-ups, debounce, and enclosure fit.
- `sndrpii2scard` detection and microphone capture with `arecord`: mono, 16 kHz,
  16-bit PCM WAV with audible, nonzero speech, including the 30-second recording
  limit and permissions for the service user. Run
  `sudo sh /opt/cloudy-pod/deploy/provision-inmp441.sh --check` for the automated
  format, size, permissions, and nonzero-level checks.
- Both-button 600 ms hold, release-to-stop, transcript review, rejection, and
  approved delivery to the selected Codex target.
- Boot-to-kiosk behavior and clean restart after power loss.
- Wi-Fi loss/recovery and read-only cached request behavior.
- Frame timing, memory, temperature, and power stability during a long run.

Voice requires the official OpenAI provider in dashboard settings. Validate it
against the deployed API before sealing the hardware image.
