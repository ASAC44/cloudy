## Prerequisites

Install Node.js 22 or newer, pnpm 11, Python 3.13, and the Supabase CLI. Link the repository to a hosted Supabase development project before applying migrations. Docker is optional and is used only by the ARM64 Pod compatibility check.

## Configure the environment

Copy the example files and replace their placeholder values:

```sh
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env.local
```

Generate a stable API encryption key with `openssl rand -base64 32`. Keep that value server-side; changing it makes saved provider credentials unreadable.

For the complete Supabase and OAuth configuration, follow [Authentication setup](/docs/development/authentication).

## Install and run

From the repository root:

```sh
./scripts/setup.sh
supabase link
supabase db push
./scripts/dev.sh
```

The script starts the dashboard on port 3000, the Hono API and its guarded rule loop on port 3001, and the desktop Pod simulator. It does not start the local Codex bridge. Pair the simulator from the dashboard with its eight-character code, then create a **New Ping**.

## Verify the workspace

```sh
./scripts/check.sh
```

The optional ARM64 compatibility check requires Docker:

```sh
./scripts/check-pod-arm.sh
```

Continue with [Pod development](/docs/pod/development) for simulator controls and local-state behavior.

Telegram application credentials are only required when a Telegram connection
is used; GitHub-only rules run without them. To send Pod voice tasks to a local
repository, follow the [Codex bridge guide](/docs/development/codex-bridge).
