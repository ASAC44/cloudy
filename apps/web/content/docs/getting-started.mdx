---

## Prerequisites

Install Node.js 22 or newer, pnpm 11, Python 3.13, the Supabase CLI, and a
Docker-compatible container runtime. Docker runs the local Supabase stack and
is also used by the optional ARM64 Pod compatibility check.

## Configure the environment

Copy the example files and replace their placeholder values:

```sh
cp apps/api/.env.example apps/api/.env
cp apps/web/.env.example apps/web/.env.local
```

The local launcher supplies the Supabase URL and API keys to both apps. You
still need a stable `CONNECTION_ENCRYPTION_KEY` in `apps/api/.env`; generate it
with `openssl rand -base64 32`. Keep it server-side because changing it makes
saved provider credentials unreadable. For hosted Supabase and OAuth
configuration, follow [Authentication setup](/docs/development/authentication).

## Install and run

From the repository root:

```sh
./scripts/setup.sh
./scripts/local.sh --reset
```

The local launcher starts Supabase, rebuilds its database from every
tracked migration, injects its local URL and keys, then starts the dashboard on
port 3000, the Hono API on port 3001, the dedicated rule worker, and the desktop
Pod simulator. It does not start the local Codex bridge. Pair the simulator
from the dashboard with its eight-character code, then create a **New Ping**.

On later runs, omit `--reset` to preserve local data:

```sh
./scripts/local.sh
```

To use a hosted development project instead, run `supabase link`,
`supabase db push`, and `./start.sh`. The local launcher never changes the
hosted environment files.

The first worker poll records existing provider items as its baseline without creating approvals. New items, or changed identities such as a new commit on a watched pull request, can create approvals on later polls.

For development, `scripts/dev.sh` enables a SQLite fallback at
`apps/api/.state/podex.sqlite` when the hosted project does not yet have the
Pod keychain-layout migration. The fallback stores only directional app layouts; all
pairing, requests, decisions, and credentials continue to use Supabase. Apply
the tracked migrations before deploying because production does not enable
this fallback. The script passes the database as an absolute
`PODEX_LOCAL_LAYOUT_DB` path so startup does not depend on the caller's working
directory.

## Verify the workspace

```sh
./scripts/check.sh
```

The optional ARM64 compatibility check requires Docker:

```sh
./scripts/check-pod-arm.sh
```

Continue with [Pod development](/docs/pod/development) for simulator controls and local-state behavior.

Telegram application credentials are only required when a Telegram connection
is used; GitHub-only rules run without them. To send Pod voice tasks to a local
repository, follow the [Codex bridge guide](/docs/development/codex-bridge).
